## Changes from baseline (run 469052)

### New features
- **`is_holiday`** added as 8th calendar feature (`num_calendar_features` 7 → 8). Uses `holidays.US()` to flag US federal holidays. Energy demand drops significantly on holidays that `is_weekend` alone doesn't capture (Christmas, Thanksgiving, July 4th, etc.).
- **Weekly demand lag** (`future_demand_lag`): replaced the zero-filled future demand placeholder in the tabular token with actual demand from 168 hours prior (same wall-clock hours, one week earlier). Extracted from `history_energy[:, :24, :]` in `adapt_inputs`; from `demand_values[start+S-168 : start+S-144]` in the dataset. `valid_starts` now requires `i + S >= 168` to guarantee the lag exists.

### Regularization
- **Dropout:** `transformer_dropout` 0.1 → **0.25**

### Optimizer
- **Cosine annealing LR scheduler** added: decays from `lr=3e-4` to `lr_min=1e-5` over 50 epochs. Previous runs used a flat learning rate.

### Sequence / architecture
- **S = 168** (set by user before training; reverted to 72 in config after this job submitted)
- Sequence length: **(168 + 24) × 26 = 4,992 tokens** (vs 2,496 in baseline)
- Trainable parameters: **889,352** (vs 876,808; extra params from larger `timestep_pos` embedding and wider `hist_embedder`/`future_embedder` input due to 8th calendar feature)

## Remaining issues

- Train/val gap still large (15 vs 54) — overfitting not fully resolved
- S=168 made epochs 2.6× slower without clear benefit over using S=72 + explicit lag
- No early stopping — last ~9 epochs after the epoch-41 best were wasted compute

## Next run plan

- S=72 with weekly lag (seq length back to 2,496; ~435s/epoch)
- Consider lowering `lr_min` to `5e-6` to reduce late-epoch noise
- Add early stopping (patience ~15 epochs)
